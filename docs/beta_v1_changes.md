# Changelog: Beta v1 — доработки по итогам бета-теста

Дата: 2026-04-13  
Источник: ТЗ «Доработка бота по подсчёту калорий (v1, по итогам бета-теста)»

---

## Задача 1. Корректный вес готового блюда (P0)

**Проблема:** бот суммировал рецептурные ингредиенты вместо оценки готового веса.
Пример: 4 сырника из 100г творога → бот называл 220г и 495 ккал, реальный вес ~150г, ~220 ккал.

**Что сделано:**
- Обновлён системный промпт текстового парсера (`bot/prompts/parse_meal.txt`):
  - Явно разграничены понятия *рецептурный вес / вес готового блюда / вес порции*
  - Добавлено правило якорного ингредиента: если пользователь назвал вес ключевого ингредиента — использовать его как основу, не суммировать ингредиенты
  - Добавлены коэффициенты потери веса при термообработке (сырники/котлеты −25–35%, мясо/рыба −30–40%, блины −20%)
  - Добавлено запрещение добавлять масло/жир молча; при подозрении — задавать уточняющий вопрос
  - Добавлены few-shot примеры: сырники из 100г творога, котлета из 200г фарша
  - Добавлено: при низкой уверенности в весе выводить «≈150г (оценка)», не как факт
- То же самое внесено в промпт vision-провайдера (`bot/prompts/vision_meal.txt`)

## Задача 2. «Исправить» = дельта-апдейт, не перезапись (P0)

**Проблема:** кнопка «Исправить» удаляла приём и просила описать заново целиком.

**Что сделано:**
- Новый AI-провайдер `bot/ai/openai_patch.py` (`OpenAIPatchProvider`)
- Новый системный промпт `bot/prompts/patch_meal.txt` — патчер принимает текущие позиции + реплику пользователя
- Новая схема `MealPatchResult` / `PatchedItem` в `bot/ai/schemas.py`
- Новый Protocol `PatchProvider` в `bot/ai/base.py`
- Новый FSM-стейт `MealStates.awaiting_patch` (старый `awaiting_correction` сохранён для совместимости)
- Кнопка «✏️ Исправить» теперь входит в патч-режим (не удаляет приём)
- Новая функция `update_meal()` в `bot/services/meal_service.py` — обновляет позиции и итоги in-place
- Поддерживаемые типы патчей:
  - Изменить вес/количество одной позиции
  - Удалить позицию
  - Добавить новую позицию
  - Заменить позицию целиком
  - Применить долю ко всему приёму («съела только половину»)
- Патч-режим активен до нажатия «✅ Верно» или `/cancel`
- Если модель не поняла правку — переспрашивает, не сбрасывая контекст

## Задача 3. Понимание долей порции: 50%, половина, треть (P1)

**Проблема:** «съела 50% из этого» считалось на 100% порции.

**Что сделано:**
- Добавлена в промпты (`parse_meal.txt`, `vision_meal.txt`) таблица нормализации долей:
  - «половина» / «пол-» / «1/2» / «50%» → ×0.5
  - «треть» / «1/3» / «33%» → ×0.33
  - «четверть» / «1/4» / «25%» → ×0.25
  - «три четверти» / «3/4» / «75%» → ×0.75
  - «две трети» / «2/3» / «66%» → ×0.67
  - «примерно половину» → ×0.5 (помечать как оценку)
- Модель применяет коэффициент к весу порции после оценки полного блюда
- Если доля написана уже после распознавания — через «Исправить»/патчер (задача 2)

## Задача 4. БЖУ для каждой позиции в выводе (P1)

**Проблема:** в ответе бота по каждой позиции указывались только калории.

**Что сделано:**
- Обновлена функция `format_meal_result()` в `bot/services/stats_service.py`
- Теперь каждая позиция выводится в формате:
  ```
  · Яйца куриные (2 шт.) — 155 ккал · Б 13 · Ж 11 · У 1
  · Рыба (1 ломтик 30г) — 60 ккал · Б 6 · Ж 4 · У 0
  ```
- Применяется и в обычном выводе, и в выводе после патча

## Задача 5. Дедупликация похожих приёмов (P1)

**Проблема:** пользователь мог случайно залогировать один и тот же приём дважды (например, фото + голос как два отдельных приёма).

**Решение (согласовано):** не дебаунс/отмена, а вопрос после второго приёма.

**Что сделано:**
- После сохранения нового приёма проверяется: есть ли другой приём этого пользователя за последние N минут (по умолчанию 10 мин, настраивается через `MEAL_DUPLICATE_WINDOW_MINUTES` в `.env`)
- Вычисляется словесная схожесть (Jaccard) между `raw_input` нового и предыдущего приёмов
- Порог схожести: 0.30 (настраивается через `MEAL_DUPLICATE_SIMILARITY_THRESHOLD`)
- Если условие выполнено — бот отправляет отдельное сообщение:
  > *Похоже на приём в 12:34 — это не повтор?*
  >
  > [Новый приём ✓]  [Повтор — удалить]
- «Повтор — удалить» → soft-delete нового приёма
- «Новый приём» → обе записи остаются
- Новая функция `get_recent_meal()` в `bot/services/meal_service.py`
- Новые callback-хэндлеры `dedup_ok:` / `dedup_delete:` в `handlers/meal.py`

## Задача 6. Vision-режим: кнопка «Уточнить» (P2)

**Проблема:** по фото бот иногда выдавал неточные цифры без возможности уточнить детали.

**Что сделано:**
- Добавлена кнопка **🔍 Уточнить** в клавиатуру результата
- Показывается при: confidence=medium или low, а также для всех photo-приёмов
- Кнопка входит в тот же патч-режим, что и «Исправить», но с другим вводным сообщением:
  > «Хочешь уточнить? Напиши — например: "блюдо было граммов 300", "там был соус", "готовила на масле"»
- Vision-промпт обновлён: явно указано задавать уточняющий вопрос при видимом соусе/заправке

## Общие изменения

### Промпты вынесены в файлы
Все системные промпты теперь в `bot/prompts/` (редактируются без деплоя):
- `bot/prompts/parse_meal.txt` — текстовый/голосовой парсер
- `bot/prompts/vision_meal.txt` — vision-парсер
- `bot/prompts/patch_meal.txt` — патчер

### Логирование
- В `openai_text.py` и `openai_vision.py` добавлены `log.debug()` вызовы:
  - промпт (количество символов + ввод пользователя)
  - сырой ответ модели
  - распарсенный JSON позиций
- В `openai_patch.py` то же самое для патч-запросов
- Уровень DEBUG — в продакшене отключать через `LOG_LEVEL=INFO`

### Конфиг
Новые параметры в `config.py` / `.env`:
- `MEAL_DUPLICATE_WINDOW_MINUTES=10` — окно дедупликации
- `MEAL_DUPLICATE_SIMILARITY_THRESHOLD=0.30` — порог схожести Jaccard
